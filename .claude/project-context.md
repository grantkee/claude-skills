## What This Software Does

This repo is a personal library of Claude Code skills and agents, synced across devices using `make` + `rsync`. Skills are slash-command prompts that give Claude domain-specific context and instructions for a targeted task. Agents are autonomous worker definitions that can be spawned as subagents in an orchestration pipeline. The primary domain they serve is the `telcoin-network` blockchain node — a Rust project combining Narwhal/Bullshark BFT consensus with EVM execution via Reth — plus Solidity smart contracts (`tn-contracts`). The repo also contains the `CLAUDE.md` workflow configuration that governs how Claude Code plans, spawns agents, and validates work across all projects.

## Architecture Overview

The repo has no build system or runtime — it is a flat collection of markdown configuration files. Two top-level directories hold all content: `skills/` (one subdirectory per skill, each containing a `SKILL.md` prompt file) and `agents/` (one `.md` file per agent). The `Makefile` provides `rsync`-based import/export between `~/.claude/skills`, `~/.claude/agents`, and this repo, making it the source-of-truth for cross-device sync. `CLAUDE.md` is the master workflow configuration; it is also synced by `make import`/`make install`. A `settings.local.json` in `.claude/` registers a `UserPromptSubmit` hook that injects a reminder to spawn the `project-context` agent whenever plan-mode keywords are detected in the user's prompt.

The orchestration model is: one orchestrator (Claude in the main context) spawns multiple named agents as subagents. Agents reference skills via the `Skill` tool. The `tn-rust-engineer` skill is a multi-phase orchestrator pipeline (task-analyzer → impl-planner → task-decomposer → rust-engineer agents → verifier → debugger on failure). Two independent security pipelines exist: a Rust/BFT pipeline (`tn-security-eval` skill, spawning 10 parallel security agents targeting telcoin-network) and a Solidity pipeline (`solidity-sentinel` agent, spawning 3 parallel analysis tracks plus `findings-verifier`). The generic `tn-nemesis-scan` skill runs an 8-phase adversarial audit pipeline against any codebase via a dedicated set of nemesis-* subagents. The verification backend (`findings-verifier`) is shared across all review-producing skills. Agents that run in the pipeline reference `.claude/project-context.md` in the *target* repo (not this one) as shared context.

## Module Map

### Skills (`skills/`)

**Development**

- `tn-rust-skills` — Reference-only skill containing all telcoin-network Rust conventions; invoked programmatically by tn-* agents, not user-invocable
- `tn-add-benchmark` — Criterion benchmark generation for hot paths
- `create-agent` — Interactive consultant for designing new agent definition files

**Testing**

- `tn-write-proptest` — Property-based test generation (proptest crate) targeting invariants
- `tn-debug-e2e` — Diagnosis of failing e2e tests from stdout/stderr logs

**Security & Auditing**
- `tn-security-eval` — Orchestrator that spawns 10 parallel security agents (tn-consensus-safety, tn-state-transitions, tn-crypto-correctness, tn-dos-vectors, tn-determinism-verifier, tn-contract-safety, tn-dependency-auditor, tn-nemesis-auditor, tn-dread-evaluator, tn-stride-threat-model), then hands all extracted findings to `findings-verifier` for independent verification and remediation

Domain patterns are git-hash-cached in `.audit/domain-patterns.md` and generated dynamically via `references/research-guide.md` (no hardcoded domain files). Spawns `nemesis-orchestrator`. Uses `references/research-guide.md` for dynamic domain discovery. The previously hardcoded `defi-patterns.md` has been removed; all domain patterns are now generated dynamically.
- `solidity-security-scan` — One-command entry point for Solidity projects: runs solidity-sentinel, solidity-nemesis, solidity-gas-architect, and optionally tn-solidity-deploy-auditor in parallel, consolidates into a unified summary
- `tn-review` — Code review and security analysis for telcoin-network Rust code; internally invokes `findings-verifier` before returning results
- `tn-review-contracts` — Code review and security analysis for tn-contracts Solidity; internally invokes `findings-verifier` before returning results

- `state-inconsistency-auditor` — Finds coupled-state bugs where one field is updated without its dependent; language-agnostic

After editing, spawns the `format-output` agent for file-specific layout.

**Tooling**
- `skill-creator` — Creates and iteratively improves skills; runs evals and benchmarks

### Agents (`agents/`)

**Orchestration**
- `project-context` — Analyzes repo architecture; writes `.claude/project-context.md`; spawned first in every plan
- `task-decomposer` — Decomposes a plan into minimal parallelizable subagent tasks; spawned after plan design; produces an Agent Assignment Summary
- `tn-debug-orchestrator` — Triages error output and routes to the right diagnostic skill

**Rust Implementation Pipeline** (spawned by `tn-rust-engineer` skill in order)
- `tn-task-analyzer` — Phase 1: analyzes task scope, affected layers, existing patterns; uses tn-rust-skills
- `tn-impl-planner` — Phase 2: plans implementation strategy, types, channels, and ordering from Phase 1 output; uses tn-rust-skills
- `tn-rust-engineer` — Phase 3: writes/patches/refactors Rust code; spawned one per task in parallel; does NOT orchestrate, plan, or verify
- `tn-verifier` — Phase 4: runs cargo check, fmt, clippy, nextest scoped to changed crates; spawns tn-debugger on failure
- `tn-debugger` — Phase 4 failure handler: classifies failures from tn-verifier, routes to debug-orchestrator, returns root cause + fix approach

**Utility Agents**
- `format-output` — Mechanical formatter spawned by the `human-writing` skill after writing/editing prose files. Applies file-type-specific layout rules (e.g., sentence-per-line in README.md files). Never changes wording — only whitespace and line-break structure. Leaf node.

**Post-Implementation Pipeline** (spawned by orchestrator after code writing)
- `tn-write-e2e-agent` — Wraps `tn-write-e2e` skill; spawned after implementation wave
- `tn-write-proptest-agent` — Wraps `tn-write-proptest` skill; spawned in parallel with tn-write-e2e-agent
- `tn-write-docs-agent` — Wraps `tn-write-crate-doc` skill; spawned after test wave
- `tn-review-agent` — Wraps `tn-review` skill; final validation before presenting results to user

**Review & Verification**
- `tn-pr-reviewer` — Standalone PR review orchestrator (manually triggered). Detects project type, spawns a code review subagent (tn-review or tn-review-contracts) and tn-security-eval in parallel, then invokes `findings-verifier` in merge mode. Distinct from `tn-review-agent` (which is a pipeline step post-implementation).
- `findings-verifier` — Shared verification backend used by tn-review, tn-review-contracts, tn-security-eval, solidity-sentinel, and tn-pr-reviewer. Operates in two modes: **Verify Mode** (spawns independent verification subagents with anti-confirmation-bias protocol) and **Merge Mode** (deduplicates two already-verified reports and computes verdicts). Never spawned independently.

**Nemesis-Scan Pipeline** (Phase -1 + 8 phases, all coordinated by `nemesis-orchestrator`, spawned by either `nemesis-scan` or `tn-nemesis-scan` skill)
- `nemesis-orchestrator` — Brain of the pipeline: owns Phase -1 domain discovery + all 8 phases, enforces core rules, manages Phase 4 feedback loop, coordinates all phase agents; accumulates cross-run memory at `~/.claude/agent-memory/nemesis-orchestrator/`
- `nemesis-strategy` — Phase -1a: reads project-context + target scope + user hints; produces a structured research plan of 3-8 domain topics organized into parallel groups; each topic becomes a `nemesis-researcher` spawn
- `nemesis-researcher` — Phase -1b (one per research topic): investigates a single domain research topic (coupled-state, adversarial-sequence, accumulator, invariant, or masking); writes a fragment to `.audit/nemesis-scan/research/RT-N.md`; spawned in parallel per group
- `nemesis-recon` — Phase 0: attacker reconnaissance; identifies attack goals, novel code, value stores, coupled value hypotheses; produces Attacker's Hit List
- `nemesis-mapper` — Phase 1: builds Function-State Matrix, Coupled State Dependency Map, and unified Nemesis Map
- `nemesis-feynman` — Phase 2: Feynman interrogation across all 7 question categories; produces per-function verdicts (SOUND/SUSPECT/VULNERABLE); also used in Phase 4 targeted re-interrogation
- `nemesis-state-check` — Phase 3: state inconsistency analysis enriched by Feynman findings; also used in Phase 4 targeted mode
- `nemesis-journey` — Phase 5: traces multi-transaction adversarial sequences using domain-specific patterns from `.audit/domain-patterns.md`; constructs concrete attack paths chaining state gaps with ordering concerns; checks path-dependent accumulators
- `nemesis-verifier` — Phase 6: verifies all CRITICAL/HIGH/MEDIUM findings, eliminates false positives via deep code tracing and optional PoC tests
- `nemesis-reporter` — Phase 7: generates final verified report and raw intermediate report from all phase artifacts

**Security Evaluation — Rust/BFT (10 parallel agents, all spawned by `tn-security-eval` skill)**
- `tn-consensus-safety` — BFT quorum logic, vote counting, leader election, DAG integrity; uses tn-harden + tn-threat-model skills
- `tn-state-transitions` — Invariant preservation, atomicity, rollback safety; uses tn-nemesis + tn-review skills
- `tn-crypto-correctness` — BLS/ECDSA signatures, hashing, key management, nonces; uses tn-review (crypto paths)
- `tn-dos-vectors` — Unbounded allocations, blocking async, amplification; uses tn-harden (blocking audit)
- `tn-determinism-verifier` — HashMap ordering, SystemTime, floating point, thread-dependent ordering; uses tn-harden (determinism)
- `tn-contract-safety` — Solidity access control, reentrancy, stake accounting, upgrade safety; uses tn-review-contracts
- `tn-dependency-auditor` — New crates, CVEs, feature flags, supply chain; uses Cargo.toml diff analysis
- `tn-nemesis-auditor` — Deep iterative business logic + state inconsistency cross-analysis; uses tn-nemesis skill
- `tn-dread-evaluator` — Attacker-perspective risk assessment, DREAD scoring, attack surface prioritization; uses tn-threat-model skill
- `tn-stride-threat-model` — STRIDE threat classification across trust boundaries; uses tn-threat-model skill

**Solidity Analysis Pipeline (generic — works on any Foundry/Hardhat project)**
- `solidity-sentinel` — Exhaustive Solidity static analysis orchestrator. Runs 3 parallel tracks (aderyn automated, slither automated, manual expert review), each independently verified through `findings-verifier`. Entry point for Solidity defensive security analysis.
- `solidity-invariant-auditor` — Extracts business logic from Solidity contracts, formalizes into mathematical properties, spawns `tn-foundry-invariant-architect` for Foundry `invariant_` test implementations. Produces `invariants.md` Property Map.
- `tn-foundry-invariant-architect` — Receives formalized invariant properties from solidity-invariant-auditor and produces compilable Foundry invariant tests with Handler contracts and ghost variables. Leaf node.
- `solidity-nemesis` — Adversarial exploit hypothesis generator for Solidity protocols. Spawns recon subagent and `solidity-invariant-auditor` in parallel, then attack surface analysis, then synthesizes multi-step exploit paths with quantified economics. Produces `nemesis.md`.
- `solidity-gas-architect` — Gas optimization analyzer. Runs forge inspect and forge snapshot, produces `gas-report.md` with before/after diffs, then mandatorily spawns `tn-solidity-change-scrutineer` for safety validation.
- `tn-solidity-change-scrutineer` — Safety validation leaf node for proposed Solidity diffs. Checks storage layout reordering, permission drift, variable shadowing, complexity spikes, and compiler version compatibility. Appends "Scrutineer Findings" to `gas-report.md`. Never proposes optimizations — validation only.
- `tn-solidity-deploy-auditor` — Foundry deployment script (`.s.sol`) security auditor. Spawns 5 parallel subagents (Key-Guard, Proxy-Sentinel, Environment-Auditor, Gas-Architect, Nemesis-Script). Delegates verification to `findings-verifier` in Verify Mode; produces `solidity-deployment-report.md`.

## Dependency Rules

- **tn-rust-engineer skill** is a multi-phase orchestrator: it sequentially spawns tn-task-analyzer → tn-impl-planner → task-decomposer (if 2+ tasks) → tn-rust-engineer agent(s) → tn-verifier. On verification failure, tn-verifier spawns tn-debugger, which routes through debug-orchestrator and returns a fix approach for re-entry to Phase 3.
- **tn-rust-skills** is never user-invocable. It is a reference skill invoked only by tn-task-analyzer and tn-impl-planner via the Skill tool to load conventions into their context.
- **nemesis-scan and tn-nemesis-scan** (skills) both spawn only `nemesis-orchestrator`. The nemesis-strategy, nemesis-researcher, and 8 phase agents are all internal to the orchestrator and must never be assigned directly by task-decomposer or the main orchestrator. `nemesis-scan` is the generic language-agnostic version; `tn-nemesis-scan` is the TN-specific version that additionally loads bundled domain reference files based on target keywords. Both use `references/research-guide.md` for dynamic domain discovery — there are no longer hardcoded domain pattern files. The `tn-nemesis` skill (separate) is the single-agent Feynman+state approach. Domain patterns are git-hash-cached in `.audit/domain-patterns.md` — the `--fresh` flag or a scope/hint change invalidates the cache and forces Phase -1 to rerun.
- **solidity-security-scan** (skill) is the one-command entry point for Solidity, consolidating solidity-sentinel + solidity-nemesis + solidity-gas-architect + optionally tn-solidity-deploy-auditor. These four are parallel, complementary paths — not a pipeline where one feeds the next.
- **findings-verifier** is the shared verification backend. It must be invoked by: `tn-review` (after Phase 2), `tn-review-contracts` (after Phase 2), `tn-security-eval` (after Phase 3 finding extraction), `solidity-sentinel` (per track, then consolidated), and `tn-pr-reviewer` (in merge mode). Never invoked independently.
- **tn-security-eval** (skill) spawns all 10 Rust/BFT security agents, then delegates all verification to `findings-verifier`. The 10 agents are internal to tn-security-eval and must never be assigned directly by task-decomposer or the orchestrator.
- **solidity-gas-architect** always spawns `tn-solidity-change-scrutineer` as the final phase. The scrutineer is a leaf node — it never spawns subagents.
- **solidity-nemesis** spawns `solidity-invariant-auditor` as a Phase 1 subagent; the invariant auditor in turn spawns `tn-foundry-invariant-architect`. Both can also be spawned standalone.
- **tn-solidity-deploy-auditor** is an independent entry point for Foundry deployment script analysis. Not part of `solidity-sentinel`'s pipeline — runs in parallel or standalone when `.s.sol` deployment scripts are the target.
- **tn-pr-reviewer** is a standalone manually-triggered agent; it is never part of the implementation pipeline. `tn-review-agent` fills that role.
- **tn-debug-orchestrator** routes to `tn-debug-e2e`, `tn-harden`, or `tn-nemesis` skills depending on failure type; it then hands off fixes to `tn-rust-engineer` agent. `tn-debugger` (distinct) is the internal pipeline failure handler within the tn-rust-engineer skill.
- **project-context** agent must be spawned before any other agent in a plan session.
- **task-decomposer** must be spawned before any implementation agents are launched. It explicitly excludes `tn-debug-orchestrator`, `project-context`, `tn-pr-reviewer`, the 10 individual Rust/BFT security domain agents, the 8 nemesis-scan phase agents, and the Solidity-specific agents from its assignment catalog.
- The `CLAUDE.md` at repo root is the authoritative workflow policy; it is installed to `~/.claude/CLAUDE.md` and also exists at the user's global `~/.claude/CLAUDE.md`.

## Key Abstractions

- **Skill** — A `SKILL.md` markdown file in `skills/<name>/` containing a slash-command prompt with domain context, conventions, and behavioral instructions. Invoked inside agents via the `Skill` tool.
- **Agent** — A `.md` file in `agents/` with YAML frontmatter (`name`, `description`, `tools`, `model`, `color`, `memory`) followed by a system prompt. Spawned as a subagent by the orchestrator or other agents.
- **tn-rust-engineer pipeline** — The 5-phase Rust development pipeline: task-analyzer (understand) → impl-planner (plan) → task-decomposer (parallelize) → rust-engineer agents (write) → verifier (validate). tn-debugger is the failure handler injected at Phase 4. This is the primary development workflow.
- **nemesis-scan pipeline** — Phase -1 (Domain Discovery) + 8-phase adversarial audit: strategy (-1a) → parallel researchers (-1b) → compile domain patterns (-1c) → recon (Phase 0) → mapper (Phase 1) → feynman interrogation (Phase 2) → state-check (Phase 3) → feedback loop (Phase 4, max 3 iterations) → journey tracing (Phase 5) → verification (Phase 6) → reporter (Phase 7). Language-agnostic. Phase -1 is skipped when the cached `.audit/domain-patterns.md` matches the current git hash, scope, and hints.
- **domain-patterns.md** — The compiled domain knowledge artifact written by `nemesis-orchestrator` Phase -1c to `.audit/domain-patterns.md`. Contains worked examples, adversarial sequences, coupled state table, and domain-specific red flags. Read by nemesis-journey, nemesis-feynman, nemesis-state-check, and nemesis-recon for domain-aware analysis. Cached by git hash; invalidated by `--fresh`, scope change, or hint change. Generated from `references/research-guide.md` format (no hardcoded domain files).
- **doc-writer pipeline** — The 7-wave sequential editing pipeline for technical prose: Wave 0 (opus, draft) → Wave 1 (vocabulary) → Wave 2 (directness) → Wave 3 (scaffolding) → Wave 4 (formatting) → Wave 5 (cleanup) → Wave 6 (review/retry) → Wave 7 (format-output). Each wave agent loads only its relevant rules from `skills/doc-writer/references/`. Waves 1-6 use sonnet; Wave 7 spawns the existing `format-output` agent. Supports `--edit-only` to skip drafting.
- **Research categories** — The five pattern categories used by `nemesis-researcher` and `nemesis-strategy`: `coupled-state` (state pairs that must stay synchronized), `adversarial-sequence` (multi-step operation paths with interleaving windows), `accumulator` (running totals that may diverge), `invariant` (correctness properties and their enforcement), `masking` (defensive patterns that hide broken invariants).
- **Canonical Finding Schema** — The structured format all review/security skills emit: `Finding ID`, `Source Agent`, `Severity`, `Title`, `Location` (file:line), `Claim` (factual assertion only — no reasoning chain), `Key Question`, `Relevant Files`, `Source`. The claim/reasoning split is the anti-confirmation-bias mechanism: `findings-verifier` subagents see only the claim, never the original analysis.
- **findings-verifier verification tiers** — Tier 1 (CRITICAL/HIGH): one agent per finding. Tier 2 (MEDIUM): batched 2-3 per agent by subsystem. Tier 3 (LOW): batched 3-5 per agent. INFO: skipped. Max 12 verification agents per invocation (overridable by caller).
- **Orchestration pipeline** — The fixed wave sequence defined in `CLAUDE.md`: project-context → task-decomposer → tn-rust-engineer skill (which internally runs its own 5-phase pipeline) → test agents (parallel) → docs agent → tn-review-agent.
- **`project-context.md`** — The shared architecture snapshot written by the `project-context` agent to `.claude/project-context.md` in the *target* repo. All downstream agents are told to read this file before working.
- **Makefile sync targets** — `import` (local → repo, additive), `import-clean` (local → repo, mirror), `install` (repo → local, additive), `clean-install` (repo → local, mirror, prompts before deleting local extras).
- **`settings.local.json` hook** — A `UserPromptSubmit` Python hook that detects plan-mode keywords and injects a reminder to spawn `project-context` first. Stored in `.claude/settings.local.json`.
- **Solidity pipeline entry points** — `solidity-security-scan` (skill, comprehensive one-command), `solidity-sentinel` (defensive canonical findings), `solidity-nemesis` (offensive exploit hypotheses), `solidity-gas-architect` (gas optimization + safety check), `tn-solidity-deploy-auditor` (deployment script layer).
- **Leaf node vs. orchestrator** — `tn-solidity-change-scrutineer`, `tn-foundry-invariant-architect`, `nemesis-reporter`, `nemesis-verifier`, `nemesis-researcher`, `findings-verifier`, and `format-output` are leaf nodes (they spawn no subagents). All others are orchestrators to varying degrees.
- **Exploit Hypothesis vs. Security Finding** — `solidity-nemesis` produces narrative exploit hypotheses with cost/profit economics (not canonical finding schema). `solidity-sentinel` produces canonical findings verified through `findings-verifier`. Complementary: sentinel is defensive (find bugs), nemesis is offensive (chain bugs into profitable attacks).
- **Deployment Layer vs. Contract Layer** — `tn-solidity-deploy-auditor` targets the deployment transaction sequence (`.s.sol` scripts); `solidity-sentinel` targets contract logic. Broadcast blocks are the key abstraction: each operation inside `vm.startBroadcast()` is a separate on-chain transaction with an exploitable gap.

## Build & Test

This repo contains no compiled code. There is no build step.

**Sync commands (all via `make`):**
```
make import          # push local ~/.claude/{skills,agents} + CLAUDE.md → repo (additive)
make import-clean    # push local → repo (mirror, deletes repo extras)
make install         # pull repo → ~/.claude/{skills,agents} + CLAUDE.md (additive)
make clean-install   # pull repo → local (mirror, prompts before deleting local extras)
make import-skill SKILL=<name>   # single skill import
make install-skill SKILL=<name>  # single skill install
make list            # show skills/agents in both locations
make diff            # dry-run showing what import would change
make help            # all targets with descriptions
```

**Prerequisites:** `rsync` must be available (standard on macOS/Linux). No other dependencies.

**No tests exist** — this is a prompt library, not a software package. Correctness is validated by running the skills/agents against real tasks in Claude Code.
